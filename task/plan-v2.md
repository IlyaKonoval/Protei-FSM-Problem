# План v2: доведение FSM-анализатора до сдаваемого состояния

## Цель

Сделать из текущей черновой реализации полноценную CLI-программу для анализа
логов FSM:

```text
app <end_states.txt> <out.csv> <log1> [<log2> ...]
```

Программа должна читать один или несколько лог-файлов, восстановить последнее
известное состояние каждой отслеживаемой FSM и записать CSV со списком FSM,
которые на конец обработанных логов не находятся в терминальном состоянии из
`end_states.txt`.

## Текущее состояние

В ветке уже есть черновая реализация в `main.cpp`:

- построчный разбор входных файлов;
- ручной парсер строк без `std::regex`;
- чтение `end_states.txt`;
- хранение состояния FSM в `unordered_map`;
- прогресс в `stderr`;
- запись CSV в файл;
- Docker/Compose/Makefile для запуска через контейнер.

Это хорошая база, но текущий код и инфраструктуру нельзя считать финальными:
есть расхождения в CLI, незафиксированная семантика времени и событий, а также
несколько рисков по корректности и воспроизводимости результата.

## Ключевые решения, которые надо зафиксировать

### 1. CLI

Финальный интерфейс:

```text
app <end_states.txt> <out.csv> <log1> [<log2> ...]
```

Причины:

- `end_states.txt` нужен для определения терминальных состояний;
- `out.csv` нужен явно, потому что результат задания описан как файл;
- несколько логов нужны для `datasets/5`.

Текущая проблема: `main.cpp` уже ожидает `out.csv`, но `Makefile` передаёт
только `END_STATES` и `INPUT`. Это надо исправить.

### 2. Семантика timestamp в CSV

В выходе поле `<таймстемп последнего изменения состояния>` должно означать
последний переход состояния, то есть строку вида:

```text
... < St: ...
```

Не надо обновлять этот timestamp на строках входящего сообщения:

```text
... > St: ...
```

Текущая проблема: в черновой реализации `last_ts` обновляется на любой
распарсенной строке FSM. Это может исказить время нахождения FSM в состоянии.

Нужно разделить поля:

- `last_state_change_ts` — timestamp последнего `< St:`;
- `global_last_ts` — максимальный timestamp среди **всех** строк лога, не только
  FSM-строк. Задание требует «ориентироваться на последний таймстемп в файле»,
  а лог содержит транспортные и payload-строки с timestamp, которые могут идти
  после последней FSM-строки. Реализация: перед вызовом `parse_line` делать
  лёгкую проверку начала строки на паттерн `YYYY-MM-DD HH:MM:SS.mmm` и обновлять
  `global_last_ts` независимо от результата парсинга;
- `current_state` — последнее состояние FSM.

### 3. Семантика последнего сообщения

Нужно явно выбрать правило для поля `<последнее сообщение в логику>`.

Предпочтительное правило:

- хранить последний event из строки `> St: ... Pr: ... EVENT`;
- при выводе писать последний известный входящий event для этой FSM.

Причина: в задании сказано “последнее сообщение в логику”, а не “сообщение,
которое непосредственно предшествовало последнему переходу”.

Текущая проблема: черновой код хранит `pending_event` и переносит его в
`last_event` только на переходе `<`. Если после последнего перехода была строка
`>` без нового `<`, такое сообщение может не попасть в CSV.

### 4. Формат длительности

Использовать единый формат:

```text
HH:MM:SS.mmm
```

Даже если длительность равна нулю:

```text
00:00:00.000
```

Причина: `test_ds1/out.csv` использует этот формат, а в `test_ds2/out.csv`
значение `0` выглядит как нестрогий пример. Формулировка задания требует время
нахождения в состоянии, а не специальное значение `0`.

Часы не ограничены 24: если лог длиннее суток, поле может содержать `49:00:00.000`
и это корректное значение. Часы накапливаются линейно, деление по модулю 24 не
применяется.

Для `test_ds2` надо отдельно зафиксировать ожидаемое расхождение:

- образец: `0`;
- расчёт по последнему timestamp файла: может быть `00:00:00.027`.

### 5. Порядок строк CSV

Вывод должен быть стабильным.

Текущая проблема: обход `unordered_map` даёт недетерминированный порядок строк.
Это неудобно для проверки, diff и защиты.

Нужно перед записью:

1. собрать зависшие FSM в `vector`;
2. отсортировать по:
   - `last_state_change_ts`;
   - `fsm_name`;
   - `id`.

Согласно правилу из п. 7 проблем, `last_state_change_ts` никогда не бывает
пустым — инициализируется при создании FSM. Поэтому пустые значения в ключе
сортировки возникнуть не могут.

### 6. CSV escaping

Сейчас поля пишутся через запятую без экранирования.

На текущих данных `name`, `state`, `event` выглядят безопасно, но лучше
добавить минимальную функцию CSV-экранирования:

- если поле содержит `,`, `"`, `\r` или `\n`, оборачивать в `"..."`;
- внутренние `"` удваивать.

Это небольшая доработка, которая закрывает форматный риск.

## Спецификация парсера

Парсер обрабатывает лог построчно. Для каждой строки:

1. **Сначала** — попытка извлечь timestamp для обновления `global_last_ts`.
   Условие: строка начинается с `YYYY-MM-DD HH:MM:SS.mmm` (23 символа,
   фиксированная позиция цифр и разделителей). Если совпадает — обновить
   `global_last_ts`, независимо от дальнейшего разбора.

2. **Затем** — попытка разобрать как FSM-строку. Обязательные элементы:

   ```text
   <ts> [<file>(<line>)] FSM: <name> id: <id>; > St: <n> <STATE> Pr: <code> <EVENT> [(<payload>)]
   <ts> [<file>(<line>)] FSM: <name> id: <id>; < St: <n> <STATE>
   ```

   - `<file>(<line>)` — опциональный токен между timestamp и `FSM:`;
     между ними может быть произвольный текст (TID, имя процесса и т.п.) —
     парсер ищет `FSM:` как якорь, не привязываясь к позиции;
   - `id:` и `<id>` разделены пробелом (`id: 0;`);
   - `St:` предшествует числовой индекс состояния, который игнорируется;
   - payload в скобках после `<EVENT>` игнорируется.

3. **Payload и прочие строки** (SIP-заголовки, JSON, пустые строки) —
   не являются FSM-строками, пропускаются после обновления `global_last_ts`.

## Проблемы текущего решения

### 1. CMake хрупкий

Текущий `CMakeLists.txt`:

- не задаёт стандарт C++;
- не задаёт `Release` по умолчанию;
- использует хрупкую проверку `if (${CMAKE_BUILD_TYPE} STREQUAL "Debug")`;
- содержит `-static` в Release-флагах — **блокер**: Ubuntu 24.04 в Docker не
  включает статическую libc/libstdc++ по умолчанию, линковка упадёт;
- не включает `-Werror`, но это можно оставить без `-Werror`, чтобы не ломать
  сдачу на разных компиляторах.

Нужно:

```cmake
project(app CXX)
set(CMAKE_CXX_STANDARD 17)
set(CMAKE_CXX_STANDARD_REQUIRED ON)
set(CMAKE_CXX_EXTENSIONS OFF)
```

Убрать `-static` из Release-флагов. Release: `-O3 -Wall -Wextra`.

И безопасно обработать пустой `CMAKE_BUILD_TYPE`.

### 2. Docker собирает только Debug

Текущий Dockerfile собирает только:

```text
/app/build/app
```

в `Debug`.

- Dockerfile собирает оба бинаря:
  - `/app/build-release/app`;
  - `/app/build-debug/app`;
- `Makefile` даёт цели `run`, `debug`, `release`, `test1`, `test2`, `test-all`.

Предпочтительный вариант: B. Он лучше соответствует требованию про Release и
Debug с санитайзерами.

### 3. Makefile не соответствует CLI

Текущий `Makefile` не передаёт `out.csv`.

Нужно добавить переменную:

```make
OUT ?= /tmp/out.csv
```

И запуск:

```make
docker-compose run --rm --build app $(END_STATES) $(OUT) $(INPUTS)
```

Также лучше поддержать `INPUTS`, а не только один `INPUT`, чтобы нормально
запускать `datasets/5`.

### 4. Прогресс засоряет stderr при редиректе

Сейчас прогресс печатается через `\r`. Это нормально в терминале, но плохо при
редиректе stderr в файл.

Низкий приоритет, но желательно:

- если `stderr` является TTY, печатать интерактивный прогресс;
- иначе печатать только короткие события начала/конца файла или не чаще одного
  раза на файл.

На Linux можно использовать `isatty(fileno(stderr))`.

### 5. Классификация FSM по префиксу может быть недетерминированной

Сейчас классы лежат в `unordered_map`, а при совпадении по префиксу выбирается
первое найденное совпадение. Итерация по `unordered_map` недетерминирована,
поэтому при нескольких подходящих классах результат непредсказуем.

Если появятся классы:

```text
A
A.B
```

то имя `A.B.1` должно матчиться с самым длинным подходящим классом `A.B`.

Нужно заменить правило на longest-prefix match с границей по точке.

**Реализация:** при загрузке `end_states.txt` сохранять имена классов в
`std::vector<std::string> sorted_classes`, отсортированный по убыванию длины.
В методе `classify` итерировать по `sorted_classes` — первое совпадение
автоматически будет самым длинным.

### 6. Кэш классификации может расти на неотслеживаемых FSM

Сейчас `class_cache` хранит результат для каждого встреченного имени FSM,
включая неотслеживаемые (значение — пустая строка). На шумных логах с большим
числом уникальных неотслеживаемых имён это раздувает память.

Нужно два контейнера вместо одного:

```cpp
mutable std::unordered_map<string, string> pos_cache;   // имя → класс
mutable std::unordered_set<string>         neg_cache;   // имена без класса
```

Это даёт O(1) для обоих случаев без хранения пустых строк-значений в map,
и при этом не теряет производительность на повторных отрицательных запросах.

### 7. Граничный случай `last_state_change_ts` при первой строке `>`

Если FSM впервые встречается на Input-строке (`> St:`), то до первого перехода
(`< St:`) поле `last_state_change_ts` не инициализировано. Вызов
`format_delta("", global_last_ts)` приведёт к некорректному результату.

**Правило:** при создании новой FSM всегда инициализировать
`last_state_change_ts` текущим `ts` строки. При последующих `< St:` —
обновлять. Это означает, что для FSM без единого перехода timestamp равен
моменту первой встречи в логе — семантически корректно.

### 8. Поле `pending_event` после рефакторинга семантики

После принятия правила «`last_event` = последний `> St:` event» (см. §3
ключевых решений) поле `pending_event` в `FsmState` становится ненужным.
Его надо удалить из структуры и из всех мест обновления, чтобы не оставалось
мёртвого кода и не путалась семантика.

Новое правило обновления `last_event`:
- при Input-строке (`>`): `st.last_event = string(p.event)` напрямую;
- при Transition-строке (`<`): `last_event` не трогать.

### 9. Хранение terminal FSM

Сейчас FSM остаётся в map даже после перехода в терминальное состояние.

Возможная оптимизация:

- если FSM перешла в terminal state, удалить её из active map;
- если позже появится новая строка по той же FSM, создать заново.

Плюс: меньше памяти на длинных логах.

Минус: надо аккуратно не потерять случай, когда terminal FSM потом снова
получает входящее сообщение.

Решение: можно оставить до финальной проверки. Если память/производительность
не проблема, не усложнять.

## Предлагаемый порядок работ

### Этап 1. Синхронизировать план, CLI и инфраструктуру

- Обновить `CMakeLists.txt`: C++17, безопасный `CMAKE_BUILD_TYPE`, Release по
  умолчанию.
- Обновить Dockerfile: решить, собираем один Debug или оба build-типа.
- Обновить `Makefile`: добавить `OUT`, `INPUTS`, цели для `test1`, `test2`,
  `test3`, `test4`, `test5`, `test-all`.
- Проверить, что команда запуска соответствует:

```text
app <end_states.txt> <out.csv> <log1> [<log2> ...]
```

### Этап 2. Исправить семантику состояния

- Разделить `last_state_change_ts` и `global_last_ts`.
- Обновлять `global_last_ts` из любой строки с валидным timestamp — до вызова
  `parse_line`.
- При создании новой FSM инициализировать `last_state_change_ts` текущим `ts`;
  обновлять его только при `< St:`.
- Хранить `last_event` по последнему `> St:` напрямую — удалить `pending_event`
  из `FsmState` и все места его использования.
- Проверить поведение, если первая встреченная строка FSM является `< St:`
  без предыдущего `>`.

### Этап 3. Сделать вывод стабильным и формально корректным

- Собрать зависшие FSM в `vector`.
- Отсортировать перед выводом.
- Добавить CSV escaping.
- Единообразно форматировать duration как `HH:MM:SS.mmm`.

### Этап 4. Улучшить классификацию end states

- Поддержать форматы:

```text
Name : STATE
Name:STATE
```

- Использовать longest-prefix match: при загрузке строить
  `sorted_classes` — `vector<string>` имён классов, отсортированный по убыванию
  длины. `classify` итерирует по нему, первое совпадение — самый длинный префикс.

```text
RegisterLogic.0.abcd -> RegisterLogic
```

- Заменить `class_cache` на `pos_cache` (`unordered_map<string,string>`) и
  `neg_cache` (`unordered_set<string>`) для раздельного кэширования
  положительных и отрицательных результатов.

- Проверить, что `Sg.SIP.UA` не ломается из-за точек внутри имени.

### Этап 5. Проверить прогресс и ошибки

- Ошибка открытия `end_states.txt` должна возвращать ненулевой код.
- Ошибка открытия входного файла должна возвращать ненулевой код.
- Ошибка открытия выходного файла должна возвращать ненулевой код.
- Прогресс должен не мешать CSV, потому что CSV пишется в отдельный файл.
- Опционально сделать TTY-aware progress.

### Этап 6. Верификация

Минимальная проверка:

```bash
make test1
make test2
```

Расширенная проверка:

```bash
make test-all
```

Что проверить:

- `test_ds1`: совпадение с `test_ds1/out.csv` с учётом LF/CRLF;
- `test_ds2`: совпадение основных полей, отдельно зафиксировать отличие по
  duration;
- `datasets/3`: ожидаемое число зависших FSM;
- `datasets/4`: проверка `RegisterLogic` и `SubscribeLogic`;
- `datasets/5`: мультифайловый вход `in1.txt in2.txt in3.txt`;
- Debug-сборка с санитайзерами завершается без диагностик.

## Итоговый критерий готовности

Решение можно считать готовым, если:

- проект собирается в Docker;
- Release-сборка проходит без ошибок компиляции;
- Debug-сборка проходит с ASan/UBSan/LSan;
- CLI соответствует `app <end_states.txt> <out.csv> <log1> [<log2> ...]`;
- CSV создаётся в указанном файле;
- порядок строк CSV стабилен;
- известное расхождение `test_ds2` по duration описано в README;
- README содержит команды сборки, запуска и проверки.
