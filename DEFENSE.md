## 1. Парсинг строки (`parse_line`, main.cpp:78)

### 1.1 Почему `string_view`, а не `string`?

`string_view` — это пара `(ptr, size)`, не владеет памятью, не аллоцирует. Когда читаем миллионы строк, каждая аллокация `string` — это `malloc`, и в сумме это ощутимо. Парсер работает над буфером `getline`, который живёт до следующей итерации цикла; всё, что парсер вытащил как `string_view`, либо используется тут же, либо копируется в `string` при вставке в мапу.

**Важная деталь:** `string_view`, возвращаемый из парсера, становится недействительным как только `line` переиспользуется в следующей итерации. Поэтому в строках 346, 352, 355 мы явно делаем `string(p.name)`, `string(p.ts)` и т.д. перед вставкой в `fsms`.

### 1.2 Что такое «якорь `FSM:`» и зачем он нужен?

Формат строки может быть:
```
2026-01-12 09:06:03.384 PrimFSM.cpp(37) FSM: Sg.SIP.UA id: 2621; > St: ...
2026-01-12 09:06:03.384 2513826 PrimFSM.cpp(37) FSM: Sg.SIP.UA id: 2621; > St: ...
```

Между timestamp и `FSM:` — опциональный TID (thread id) и обязательный `<file>(<line>)`. Их длина переменная. Поэтому мы не можем просто взять `substr(offset, n)` — нужно найти `FSM:` через `find`.

### 1.3 Что если подстрока `FSM:` встретится в payload (например, в SIP-теле)?

Например, в ds2 есть payload-строки вроде `SIP/2.0 180 Ringing`. Если бы какая-то такая строка содержала `FSM:`, парсер всё равно вернул бы `false` — потому что дальше проверяется цепочка: наличие `id:`, наличие `;`, наличие `>` или `<`, наличие `St:`. Вероятность случайно собрать всю цепочку — нулевая.

Но есть ещё одна защита: `has_ts_prefix` отсекает строки без timestamp. Payload-строки SIP-тела timestamp не имеют, поэтому до `parse_line` вообще не доходят как отдельные — они идут под своим timestamp-заголовком? Нет. Тут тонкость: в ds2 SIP-тело реально идёт отдельными строками без timestamp в начале. Они не пройдут `has_ts_prefix` и отвалятся на первой же проверке.

### 1.4 Почему `std::from_chars`, а не `std::stoi`/`std::atoi`?

- `std::stoi` бросает `std::invalid_argument`/`std::out_of_range` — исключения в горячем цикле ведут к unwind-стоимости.
- `std::atoi` не сообщает об ошибке — на битом input молча вернёт 0.
- `std::stoi` требует `std::string`, то есть скопировать `string_view` — аллокация.
- `std::from_chars` работает с парой указателей, noexcept, возвращает `errc` — проверяешь `if (ec != std::errc()) return false;` и всё.

Плюс `from_chars` не зависит от локали: `std::stoi` в экзотических локалях может парсить числа с разделителями тысяч.

### 1.5 Что делает блок «пропускаем числовой индекс перед именем состояния»?

```cpp
while (p < line.size() && line[p] >= '0' && line[p] <= '9') ++p;
while (p < line.size() && line[p] == ' ') ++p;
```

Формат `St:` — это `<номер> <имя>`: `< St: 5 PRE_ANSWER`. Нам нужно **имя**, а не номер. Номер — это внутренний индекс FSM-движка, он может меняться между версиями; имя — стабильный идентификатор состояния.

Если бы в задании нужны были номера, хранили бы их. Но в выходном формате ожидается именно имя состояния (см. эталонный `out.csv`).

### 1.6 Как извлекается `event`?

Формат входящей строки:
```
> St: 2 INIT_WAIT_STORAGE_CONF Pr: 38405:11 SIP_TR_SUBSCRIBE_IND (Event: protei-test-event Expires: 120)
```

Нас интересует `SIP_TR_SUBSCRIBE_IND` — это имя события. Идёт после `Pr: <code>`, заканчивается на пробел/скобку/конец строки.

```cpp
size_t pr_pos = line.find("Pr:", p);
if (pr_pos == string_view::npos) { out.event = {}; return true; }
```

Если `Pr:` отсутствует, событие пустое — это валидно, просто в CSV будет пустая колонка. Не ошибка парсинга.

### 1.7 А если формат вдруг изменится (например, добавят новое поле)?

Парсер устойчив к вариациям между timestamp и `FSM:` (TID опционален), устойчив к пробелам (везде есть `while (... == ' ') ++p`), устойчив к концу строки (везде проверка `p < line.size()`). Если изменится сам формат `FSM:`/`id:`/`St:` — парсер честно отдаст `false` на новых строках.

## 2. Классификация FSM (`EndStates::classify`, main.cpp:186)

### 2.1 Зачем longest-prefix match?

Имя в логе: `SubscribeLogic.0.61bf91b400000002`.
Имя в `end_states.txt`: просто `SubscribeLogic`.
Между ними — instance id через `.` как разделитель.

Нам нужно, чтобы любой конкретный экземпляр (`SubscribeLogic.0.abc`, `SubscribeLogic.1.def`) относился к классу `SubscribeLogic`.

### 2.2 Зачем граница по точке (`name[cls.size()] == '.'`)?

Представь, что в `end_states.txt` есть класс `Sg.SIP` и класс `Sg.SIP.UA`. Имя в логе — `Sg.SIP.UA.0.123`.

Без границы по точке наивный префикс-матчер может решить, что `Sg.SIP` подходит (он ведь префикс), но правильный класс — `Sg.SIP.UA`. Сортировка по длине тут помогает: сначала проверяется `Sg.SIP.UA` (длиннее), только потом `Sg.SIP`. Граница `.` гарантирует, что `Sg.SIP` не захватит `Sg.SIPExtra`, если бы такой был.

### 2.3 Зачем сортировка `sorted_classes` по убыванию длины?

```cpp
std::sort(sorted_classes.begin(), sorted_classes.end(),
          [](const string& a, const string& b) { return a.size() > b.size(); });
```

Линейный поиск по отсортированному списку: первое совпадение — самое длинное, то есть наиболее специфичный класс. Это и есть «longest-prefix» на ровном месте.

Альтернатива — trie по точкам, но для ~десятка классов это overkill.

### 2.4 Два кеша: `pos_cache` и `neg_cache`. Зачем два?

`pos_cache: name → class` — если имя уже классифицировали, возвращаем сохранённый класс.
`neg_cache: set<name>` — если имя НЕ подошло ни под один класс, запоминаем, чтобы не проходить по `sorted_classes` ещё раз.

Почему не один `unordered_map<string, string>` с пустой строкой как «не найдено»?
- В `pos_cache` значения — это указатели внутрь `sorted_classes`/`by_class`, можно вернуть `const string&`.
- Для «не найдено» нужна была бы ещё одна ветка логики.
- Два отдельных контейнера читаются как явная семантика: «да, это класс X» vs «это точно не наш FSM».

### 2.5 Зачем `mutable` на кешах?

`classify` вызывается от `const EndStates&` — контрактно он ничего не меняет в конфигурации. Но кеш внутри — это implementation detail оптимизации, он меняется при каждом вызове. `mutable` разрешает модификацию в const-методе — это канонический паттерн для memoization.

### 2.6 Что, если имя в логе встречается миллион раз — кеш не разрастётся?

Число уникальных **имён** (не экземпляров!) — это число активных FSM в системе. У каждого экземпляра имя вроде `SubscribeLogic.0.abc`, но конкретных `abc` может быть тысячи. Кеш хранит по одной записи на уникальное имя. Для 8 GB RAM и нескольких тысяч экземпляров — капля.

### 2.7 Почему `by_class` — это `unordered_map<string, unordered_set<string>>`, а не, скажем, `flat_map`?

У нас STL-only по требованию задания. `flat_map` появится в C++23 и только в форме `std::flat_map`. В C++17 `unordered_*` — стандартный выбор.

Для ~десятка классов и ~десятка терминальных состояний в каждом разница в производительности неощутима; `unordered_*` выбран потому что «он и так O(1)».

## 3. Ключ и хэш FSM (`FsmKey`/`FsmKeyHash`, main.cpp:23–38)

### 3.1 Почему ключ — пара `(name, id)`, а не одна строка `name:id`?

- Не нужно конкатенировать и аллоцировать ещё одну строку.
- Сравнение `int64` + `string` быстрее, чем сравнение большой строки.
- Хэш можно комбинировать из готовых хэшей.

### 3.2 Откуда взялась константа `0x9e3779b97f4a7c15`?

Это `2^64 / φ`, где φ — золотое сечение. Используется в Boost для `hash_combine`: даёт равномерное распределение бит при комбинировании двух хэшей. Формула:

```cpp
h ^ (g + 0x9e3779b97f4a7c15ULL + (h << 6) + (h >> 2));
```

Смысл: подмешать хэш `g` к `h` так, чтобы равномерно распределить по всем битам. Сдвиги делают результат нелинейным относительно исходных битов.

### 3.3 Почему `operator==` сравнивает `id` первым?

```cpp
return id == o.id && name == o.name;
```

`int64 == int64` — одна инструкция CPU.
`string == string` — сначала сравнение длины, потом `memcmp`.

При коллизии хэша (редкой) почти всегда отличаются id, и short-circuit `&&` не доходит до сравнения имён. Это микро-оптимизация, но в горячем пути она не бесплатна.

### 3.4 Почему `int64_t`, а не `uint64_t`?

`from_chars` по умолчанию парсит знаковые числа; id в логах неотрицательные, но представимы в int64 без потерь. Знаковый тип безопаснее при случайных арифметических операциях (нет wrap-around surprise).

## 4. Основной цикл (`main.cpp:326`)

### 4.1 Почему построчное чтение через `getline`, а не mmap?

- STL-only (требование задания) — `mmap` это POSIX, не STL. Хотя мы уже используем `<sys/stat.h>` и `<unistd.h>` для `stat` и `isatty`, но это минимальный системный интерфейс.
- Построчный API проще: не надо вручную искать `\n` через блоки, обрабатывать границы mmap-страниц, размапливать.
- Для 8 GB RAM / ~нескольких сотен МБ логов — I/O не бутылочное горло.

mmap дал бы выигрыш при гигабайтных логах, но усложнил бы код в разы.

### 4.2 Почему `line.reserve(512)`?

Типичная длина FSM-строки лога — 150–300 байт. Payload-строки могут быть длиннее. `reserve(512)` выделяет буфер сразу, чтобы `getline` не делал `realloc` на первых сотнях строк. Это оптимизация под «прогрев», её эффект мал, но бесплатен.

### 4.3 Почему `global_last_ts` обновляется из **любой** строки, а не только FSM-строки?

Задание: «время нахождения логики в данном состоянии — ориентироваться на последний таймстемп в файле».

В файле встречаются:
1. FSM-строки (`> St:`, `< St:`).
2. Не-FSM строки с timestamp (логирование SIP-пакетов, и т.п.).
3. Payload-строки без timestamp (продолжение SIP-тела).

Последняя строка файла может быть не FSM-строкой. Если бы мы брали максимум только из FSM-строк, то недооценили бы длительность «висения».

Пример из ds2: последняя FSM-строка в 09:16:45.945, но в файле есть ещё payload-строки после, и последняя имеет timestamp позже. Эталонный `out.csv` там даёт `0` (подозрительно), а мы — `00:00:00.027`. Расхождение зафиксировано в README.

### 4.4 Лексикографическое сравнение timestamp — это корректно?

Да. Формат `YYYY-MM-DD HH:MM:SS.mmm` имеет фиксированную ширину всех полей и упорядочен по убыванию значимости. Лексикографическое сравнение байтов эквивалентно хронологическому. Это классический трюк — ISO 8601 специально так спроектирован.

### 4.5 Почему порядок: сначала `parse_line`, потом `is_tracked`?

- `parse_line` быстро отсекает большинство строк (не-FSM, payload).
- `is_tracked` требует имя, которое даёт парсер.
- Кеш `is_tracked` делает его O(1) амортизированно.

Инверсия была бы бессмысленной: мы не знаем имени до парсинга.

### 4.6 Первое упоминание FSM — почему `last_event` заполняется только при `Input`?

```cpp
if (it == fsms.end()) {
    FsmState st;
    st.last_state_change_ts = string(p.ts);
    st.state                = string(p.state);
    if (p.kind == LineKind::Input) st.last_event = string(p.event);
    fsms.emplace(std::move(key), std::move(st));
    continue;
}
```

`last_event` — это поле «последнее входящее сообщение». Если первое упоминание FSM — это `< St:` (переход), то никакое сообщение её ещё не пришло (вернее, пришло раньше, но мы его не видели). Заполнять `last_event` из состояния бессмысленно.

При этом `last_state_change_ts` ставится всегда — у нас нет «старого» ts, чтобы использовать иначе, и задание требует этот timestamp непустым.

### 4.7 При `Input` обновляется и `state`, и `last_event`. Зачем обновлять state, если меняет его только `<`?

Строка `> St: 5 PRE_ANSWER Pr: ...` — это «сообщение пришло, текущее состояние FSM — `PRE_ANSWER`». То есть сам факт прихода сообщения подтверждает, в каком состоянии находится FSM в этот момент. Если между `<` и следующим `>` прошло время и что-то случилось (потенциально), `>` даёт нам актуальную точку.

На практике `state` из `>` обычно совпадает с последним `<`, но делать ассерт мы не стали — лог может содержать пропуски/битые строки, и перезапись в `Input`-ветке работает как self-healing.

### 4.8 При `Input` не обновляется `last_state_change_ts` — а что если между двумя `<` пришло сообщение?

`last_state_change_ts` — время **последней смены состояния**. Входящее сообщение состояние не меняет (если после него не идёт `<`). Соответственно обновлять ts не надо.

Если после `>` идёт `<` с новым состоянием — `<` перезапишет и ts, и state. Если `<` идёт с тем же состоянием, ts всё равно обновится — формально «FSM снова вошла в это состояние» (хотя оно то же).

## 5. Форматирование длительности (`format_delta`, main.cpp:219)

### 5.1 Почему `timegm`, а не `mktime`?

- `mktime` интерпретирует `struct tm` как локальное время и применяет таймзону процесса. Результат зависит от `TZ` окружения.
- `timegm` трактует как UTC — полностью детерминирован.
- Для разности двух timestamp абсолютное значение неважно (таймзона сократится), но:
  - `mktime` может задуматься над DST и подвинуть результат на час.
  - `timegm` не подвержен DST вообще.

### 5.2 Что такое `as_ms` и зачем?

Лямбда превращает строку `"2026-01-12 09:06:03.384"` в количество миллисекунд от эпохи. Потом две разности — разность в мс, а оттуда уже `h/m/s/ms`.

Альтернатива — написать свою арифметику дат (високосные годы, разная длина месяцев). `timegm` даёт это бесплатно.

### 5.3 Почему часы не ограничены 24?

Формат `HH:MM:SS.mmm` традиционно про время суток, но здесь это **длительность**. Если FSM висит 2 суток, корректный вывод — `48:00:00.000`, не `00:00:00.000` (обнуление) и не `2 00:00:00.000` (лишнее поле дней).

`%02lld` с long long позволяет вывести любое число часов — даже 999999.

### 5.4 Что если `b < a`?

```cpp
int64_t dms = as_ms(b) - as_ms(a);
if (dms < 0) dms = 0;
```

В корректном логе `global_last_ts >= last_state_change_ts` всегда. Но если данные испорчены (например, переход зафиксирован позже последнего ts файла из-за гонки в логгере), мы не вылетаем с ассертом — просто выдаём `00:00:00.000`.

Это defensive programming: программа не должна падать на битом input.

### 5.5 Почему в `as_ms` используется `stoi`, а не `from_chars`?

- Вызов один раз на зависшую FSM (их обычно единицы-десятки).
- Строка уже скопирована в `std::string` (в `FsmState`).
- Читается проще.
- Разница в производительности — микросекунды на весь запуск.

Это сознательный trade-off: в горячем цикле парсинга — `from_chars`, в холодном выводе — `stoi`.

### 5.6 Что делает `snprintf` — можно ли было через `std::format`?

`std::format` — C++20, у нас C++17 (указано в CMakeLists). В C++17 выбор — `snprintf` или `stringstream`. `snprintf` быстрее, компактнее, без аллокаций (буфер на стеке).

`char buf[32]` хватает: `HH:MM:SS.mmm\0` = 13 байт, даже с 6-значными часами — 17 байт. Запас есть.

## 6. CSV-вывод (`csv_field`, main.cpp:253; запись — main.cpp:398)

### 6.1 Правила экранирования

По RFC 4180:
- Поле с `,`, `"`, `\r`, `\n` — оборачивается в `"..."`.
- Внутренние `"` внутри поля удваиваются: `"he said ""hi"""`.
- Обычное поле — без кавычек.

```cpp
if (s.find_first_of(",\"\r\n") == string::npos) return s;
```

Fast path: если в поле ни одного спецсимвола — возвращаем как есть. В 99% случаев имена FSM и состояний — ASCII без запятых.

### 6.2 А UTF-8 надо как-то обрабатывать?

Нет. CSV не требует экранирования non-ASCII байтов — они проходят насквозь. Наши поля в ASCII, но даже если бы были кириллицей — работало бы корректно.

### 6.3 `key.id` выводится без `csv_field` — это безопасно?

Да. `int64_t` сериализуется как цифры + опциональный знак. Ни запятых, ни кавычек быть не может.

### 6.4 `key.id` выводится в поток `out << key.id` — как это работает?

`ofstream` наследует `operator<<(int64_t)` от `ostream`. Никакой магии, стандартный способ.

### 6.5 Почему финальная сортировка `vector<pair>` перед записью?

`unordered_map` даёт **недетерминированный** порядок. Между:
- запусками одной и той же сборки (у `unordered_map` есть случайный seed хэша на GCC при `-D_GLIBCXX_USE_RANDOMIZATION` — впрочем, у нас этого нет),
- версиями STL,
- разными компиляторами (libstdc++, libc++, MSVC STL),

порядок будет разным. Это мешает diff-проверкам с эталоном.

Сортировка по `(ts, name, id)` даёт устойчивый вывод: одинаковый input → одинаковый output, байт-в-байт.

### 6.6 Почему сортировка именно по `(ts, name, id)`?

- `ts` первым — хронологический порядок «когда застряли» интуитивен для аналитика.
- `name` — группирует FSM одного класса рядом.
- `id` — финальный тай-брейкер, уникален в пределах имени.

### 6.7 Что если у двух FSM одинаковый `ts` до миллисекунды — порядок всё равно детерминирован?

Да, тай-брейкеры по `name` и `id` гарантируют total order. Одинаковых `(ts, name, id)` быть не может — это был бы один и тот же ключ в мапе.

## 7. Прогресс и I/O

### 7.1 Почему прогресс только в TTY (`isatty(stderr)`)?

- Интерактивный запуск (`./app ...` в терминале) — хочется видеть `\r[progress] ... 50%`.
- Redirect в файл (`./app ... 2>log`) — `\r` создаст мусор в файле, и 100 строк прогресса загрязнят лог.
- Docker без TTY — то же самое.

`isatty(fileno(stderr))` — POSIX API для определения, подключён ли stderr к терминалу. Результат кешируется (stderr за время процесса не меняется).

### 7.2 Почему прогресс в stderr, а не stdout?

- stdout зарезервирован под данные (на случай, если захотим писать CSV в stdout).
- stderr — диагностика, прогресс, ошибки — канонически.
- При `2>/dev/null` прогресс пропадёт, а результат — останется.

### 7.3 Как считается процент?

```cpp
int64_t bytes = 0;
while (std::getline(f, line)) {
    bytes += static_cast<int64_t>(line.size()) + 1;  // +1 за '\n'
    int pct = static_cast<int>((bytes * 100) / total);
    if (pct != last_pct) { print_progress(...); last_pct = pct; }
}
```

Печатаем только при **изменении** процента — не на каждой строке. Иначе `fprintf` на каждой строке замедлит всё в разы (syscall write).

### 7.4 Точность `+ 1` за newline?

Предполагаем `\n` (LF). Если лог в CRLF, каждая строка недоучтёт 1 байт → общий прогресс будет показывать `99%` в конце вместо 100. Это косметика, данные не страдают.

## 8. Управление памятью

### 8.1 Как гарантируем «помещаемся в 8 GB RAM»?

Что хранится в памяти:
1. `by_class: map<string, set<string>>` — размер `end_states.txt`, килобайты.
2. `sorted_classes: vector<string>` — копии имён классов, ещё килобайты.
3. `pos_cache`/`neg_cache` — по одной записи на уникальное имя FSM-экземпляра.
4. `fsms: map<FsmKey, FsmState>` — основной потребитель. Каждая запись — имя + 3 строки + id.

Если активных FSM ~1 млн и каждая занимает ~200 байт → 200 МБ. Даже 10 млн — 2 ГБ, вписываемся.

**Что НЕ хранится**: лог целиком. Он потоково проходит через `getline`.

### 8.2 А если в логе 100 млн FSM?

Тогда упрёмся. Решения:
- Spill на диск (sorted temp files, merge).
- Ограничить количество отслеживаемых классов (задание даёт `end_states.txt`).

Но в рамках задания таких объёмов нет — 8 ГБ на терабайтный лог будет достаточно, пока активных FSM не слишком много. В реальных логах большинство FSM быстро приходят в терминал и остаются «не застрявшими», но всё равно живут в мапе до конца обработки.

### 8.3 Потенциальная оптимизация: удалять завершённые FSM из мапы?

Мы могли бы на каждом `<` проверять `is_terminal` и если да — удалять из `fsms`. Тогда мапа содержала бы только активных.

Но это ломает корректность: если FSM вошла в терминал, а потом снова в нетерминал (reinit сценарий), мы потеряем историю. В задании такой случай не исключён.

Компромисс: не удаляем, но памяти хватает на реалистичных объёмах.

### 8.4 Санитайзеры — какие?

```cmake
-fsanitize=leak,address,undefined
```

- **ASan** (`address`) — use-after-free, buffer overflow, double-free.
- **UBSan** (`undefined`) — signed overflow, null deref, misaligned access, deleted virtual call.
- **LSan** (`leak`) — утечки памяти (при выходе из программы).

Запуск: `make native-debug` → `./build-debug/app ...`. На всех датасетах чисто.

### 8.5 Есть ли UB в коде?

Осознанно — нет. Проверки границ везде (`p < line.size()`), знаковые переполнения не могут случиться (int64 для тысячелетий достаточно). UBSan чист.

## 9. Сборка

### 9.1 Почему два билд-типа?

Требование задания:
- **DEBUG** — с санитайзерами (через `-DCMAKE_BUILD_TYPE=Debug`).
- **RELEASE** — без санитайзеров.

### 9.2 Флаги Release: `-O3 -Wall -Wextra`

- `-O3` — максимальная оптимизация (inlining, vectorization, loop unrolling).
- `-Wall -Wextra` — все разумные варнинги. Без варнингов — критерий оценки.

Не используем `-march=native` — бинарь должен запускаться на любой машине.

### 9.3 Флаги Debug: `-fsanitize=... -g -Wall -Wextra`

- `-fsanitize=leak,address,undefined` — санитайзеры.
- `-g` — debug-символы для gdb/сatcher stack trace у санитайзеров.
- Оптимизации нет (`-O0` по умолчанию в Debug).

### 9.4 Почему `CMAKE_CXX_STANDARD 17`?

- Нужен `std::from_chars` (C++17).
- Нужен `std::string_view` (C++17).
- C++20 дал бы `std::format`, `std::span`, но у нас их и так не хватает в STL Ubuntu 24.04 старого gcc? Нет, gcc 13 их имеет. Просто C++17 достаточно, не было смысла поднимать.

### 9.5 `CMAKE_CXX_EXTENSIONS OFF` — что это?

Отключает GNU-расширения (например, `typeof`, VLA). Гарантирует, что код компилируется чистым стандартом C++17 — портабельность.

### 9.6 Docker — зачем?

1. **Воспроизводимая сборка.** Ubuntu 24.04 из коробки, точный список пакетов (`build-essential`, `cmake`, `ninja-build`).
2. **Проверка под Linux.** Задание требует Linux; проект разрабатывается на Windows, Docker даёт реальный Linux-билд.
3. **Изолированный тест.** `make run ENDS=... INPUTS=...` запускает в контейнере — не трогает хост.

### 9.7 Почему `ninja-build` установлен, но не используется?

В нашем `cmake` вызове мы не передаём `-G Ninja`, то есть по умолчанию идёт Make. Ninja был бы быстрее, но разница на одном `main.cpp` — миллисекунды. Оставили в образе на случай ручных экспериментов.

## 10. Тонкие места и защита от «каверзных» вопросов

### 10.1 Что если в `end_states.txt` один класс имеет несколько терминальных состояний?

Формат:
```
Sg.SIP.UA : NUL
Sg.SIP.UA : TERMINATED
```

Обе строки парсятся, обе добавляются в `by_class["Sg.SIP.UA"]`. `is_terminal` проверяет вхождение текущего состояния в set — любое из этих двух засчитывается как терминал.

### 10.2 Что если в `end_states.txt` строка без `:` или пустая?

```cpp
if (sv.empty() || sv[0] == '#') continue;
size_t colon = sv.find(':');
if (colon == string_view::npos) continue;
...
if (name.empty() || st.empty()) continue;
```

Пропускается без ошибки. Комментарии через `#` тоже поддерживаются.

### 10.3 Что если одна FSM инициализируется несколько раз (id сбрасывается на 0)?

В нашей модели `(name, id)` — глобально уникальный ключ. Если после терминала та же FSM появляется снова с тем же id — мы перезаписываем её состояние. Это правильное поведение для FSM-движка, где id переиспользуется (например, pool).

Если в логе `< St: NUL` (терминал), а потом `> St: 0 NUL ...` снова, мы:
- Увидим `< St: NUL` → `state = NUL`, `last_state_change_ts` обновится.
- Увидим `> St: 0 NUL Pr: ...` → обновим `last_event`, `state` тот же.
- На выходе FSM НЕ попадёт в stuck (она в `NUL`, терминал).

Всё корректно.

### 10.4 Что если timestamp в файле идут не в порядке возрастания?

`global_last_ts` всё равно берёт максимум: `if (ts > global_last_ts) global_last_ts = ...`. Но `last_state_change_ts` обновляется **без** сравнения: мы верим, что `<` идут в хронологическом порядке в рамках одной FSM.

Если нет — поле будет содержать timestamp последнего `<`, не обязательно хронологически последний. На практике логи пишутся последовательно, такой гонки не бывает.

### 10.5 Несколько входных файлов — как объединяются?

Файлы обрабатываются последовательно, один и тот же `fsms` общий для всех. `global_last_ts` тоже общий — берётся максимум по всей сессии.

Семантически это как один большой лог: FSM, начавшая в file1 и продолжившая в file2, корректно прослеживается. Это видно в ds5.

### 10.6 Что если два файла дают один `global_last_ts`, но внутри одного файла FSM застряла давно?

`duration = global_last_ts - last_state_change_ts`. Если FSM застряла в file1 в 08:00, а file2 заканчивается в 20:00, `duration = 12:00:00.000`. Это корректно: «FSM висит с момента последнего перехода до конца всей сессии».

### 10.7 Почему в `main` нет `try/catch`?

- `std::from_chars` не бросает.
- `std::ifstream`/`std::ofstream` не бросают по умолчанию (exceptions off).
- `stoi` в `format_delta` **может** бросить на битом timestamp, но мы его уже провалидировали в `has_ts_prefix`. Там жёсткие позиции `-`, `:`, `.`.

На practice `bad_alloc` единственное, что могло бы сработать; его мы не ловим, программа упадёт с сообщением от libc — что разумно при исчерпании памяти.

### 10.8 Конкурентность?

Нет. Одна нить. Это намеренно:
- Обработка одного файла последовательна (порядок состояний важен).
- Параллелизация по файлам требует мерджа таблиц — сложность несопоставима с выигрышем на 3 файлах.
- Блокировок нет — нет и deadlock-ов.

### 10.9 Тестирование

В проекте нет unit-тестов (Google Test или подобного). Тестируется через:
1. Датасеты ds1–ds5 с ожидаемыми CSV — сравнение через diff (кроме ds2, где задокументировано расхождение формата).
2. Санитайзеры на debug-сборке — ловят memory errors.
3. Сборка без варнингов с `-Wall -Wextra`.

Причина: задание не требовало unit-тестов, а писать их под одну функцию монолита — перерасход времени. Интеграционные прогоны ds1–ds5 покрывают все пути.

### 10.10 Что бы я улучшил, если бы было больше времени?

1. Unit-тесты для `parse_line` на edge cases (пустая строка, битый id, unicode).
2. Benchmark на гигабайтном логе для честных чисел «скорости».
3. mmap для больших файлов (x2–x3 ускорение I/O).
4. Возможно, разбить `main.cpp` на отдельные файлы (parser.hpp, end_states.hpp) — но для 413 строк overkill.
5. Вывод статистики: сколько всего FSM-строк обработано, сколько отброшено, сколько классов матчились.

## 12. Возможные «стресс-вопросы» эксперта

**«Покажите, где в коде вы обрабатываете edge case X?»** — знать точные номера строк (см. карту в секции 0).

**«А если я подам файл на 100 ГБ?»** — скорость I/O линейная, память константная (растёт только число активных FSM). Проблема только в том, что `stat` даёт `int64_t` — потолок в эксабайтах.

**«Покажите вывод санитайзеров на конкретном датасете.»** — `make native-debug && ./build-debug/app testdata/ds5/end_states.txt /tmp/x.csv testdata/ds5/in1.txt testdata/ds5/in2.txt testdata/ds5/in3.txt`. Чистый exit code 0, санитайзеры молчат.

**«Покажите, как ведёт себя на битом input.»** — передать `/etc/passwd` как log. Парсер вернёт `false` на всех строках, программа завершится с `0 stuck FSM(s)`, без падения.

**«Почему не вынесли парсер в отдельный класс с методом `parse()`?»** — для 72 строк парсера overhead класса и его тестового API не оправдан. Свободная функция на `string_view` — проще, тестируема, inline-friendly для компилятора.

**«Почему `unordered_map` из STL, а не `absl::flat_hash_map` или `boost::unordered`?»** — задание требует STL-only. Боросl/abseil — внешние зависимости, запрещены.
