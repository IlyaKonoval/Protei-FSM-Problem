# План: анализатор зависших FSM (задание Protei)

## Контекст

Задание (`Задание.pdf`): разобрать логи FSM Protei и выдать CSV со списком
«зависших» FSM — тех, что на конец лога не находятся в терминальном
состоянии, заявленном в `end_states.txt`. Требования: сборка под Linux через
CMake, Release — `-O3 -static`, Debug — с ASan/UBSan/LSan, только STL и
системные вызовы, экономия памяти, вывод прогресса, входные файлы через
аргументы CLI.

За эту сессию реализация уже сделана примерно на 90% и проходит предоставленные
тесты. Данный план описывает только то, что осталось.

## Текущее состояние

- `main.cpp` — одно-файловая реализация
  - Ручной разбор строк в `parse_line()` (без `std::regex`), понимает обе
    формы: `> St:` (входящее сообщение) и `< St:` (переход); корректно
    обрабатывает опциональное поле TID между таймстемпом и `PrimFSM.cpp(...)`,
    которое встречается в реальных логах
  - Структура `EndStates`: читает `end_states.txt` (форматы `Имя : Состояние`
    и `Имя:Состояние`), сопоставляет имена инстансов по префиксу с границей
    по точке (`RegisterLogic.0.abcd` → класс `RegisterLogic`). Кэш по имени
    инстанса — линейный поиск класса выполняется один раз на инстанс
  - FSM, чьи имена не перечислены в `end_states.txt`, пропускаются на этапе
    парсинга (ограничивает рост map’а на больших продуктовых логах)
  - Прогресс печатается в `stderr` с перезаписью строки через `\r`; размер
    входного файла берётся через `stat()`
  - Длительность форматируется как `HH:MM:SS.mmm` (`format_delta`
    через `timegm` / `_mkgmtime`)
- `CMakeLists.txt` — C++17, Release по умолчанию, Debug с
  `-fsanitize=address,undefined,leak -Wall -Wextra`
- `Dockerfile` + `.dockerignore` — ubuntu:22.04, собирает обе конфигурации

## Открытые вопросы (нужно решение)

1. **Формат длительности для нуля.** В `test_ds1/out.csv` — `00:00:00.000`,
   в `test_ds2/out.csv` — `0`. Образцы несогласованы. Сейчас я всегда
   печатаю `HH:MM:SS.mmm`. Для `test_ds2` мой результат `00:00:00.027`:
   последний `<` у FSM — `.918`, последний таймстемп в файле — `.945`,
   разница 27 мс, что буквально соответствует формулировке в задании
   («ориентироваться на последний таймстемп в файле»). Предлагаю оставить
   как есть — `0` в образце выглядит как сокращённая запись нуля, а не как
   требование формата.

2. **Окончания строк.** Релизный Linux-бинарник пишет LF. Образцы фикстур —
   CRLF (созданы на Windows). Ничего не правлю — для Linux-сборки LF
   корректен.

   инструкции по сборке и запуску Release/Debug, однострочник для Docker,

2. **Глушить прогресс, когда `stderr` не TTY** — опционально. Сейчас
   обновления через `\r` засоряют файл при редиректе. Низкий приоритет — в
   задании требуется только, чтобы прогресс печатался.

3. **Финальная проверка**: пересобрать Release + Debug в Docker, прогнать
   все 5 фикстур, сверить с `out.csv` где он есть, подтвердить коды выхода
   и чистые запуски с санитайзерами. Код менять не планирую — это
   приёмочный прогон перед сдачей.

Не в рамках: оптимизаций сверх уже сделанных нет; дополнительных тестов
писать не планирую; потоковый/чанковый вывод прогресса — только если
что-то упадёт на фикстурах.

    /data/datasets/5/in{1,2,3}.txt
